JS-DOM Bridge Gap — Why React & Next.js Pages Are Broken

Status: v0.1.0-beta blocker. This is the single largest reason the browser cannot yet render modern JavaScript-heavy sites.

The Symptom

When loading a Next.js or React application, the browser shows either:

  • A completely blank viewport
  • A static shell with no interactive content
  • Broken layout with missing text/images

Static HTML + CSS pages render correctly. The difference is JavaScript execution that mutates the DOM.

The Root Cause

The file source/webCore/scripting/js_dom_bridge.hpp contains only declarations:

class JSDOMBridge {
  void initialize(Runtime::VM* vm, WebCore::DOMDocument* document);
  Runtime::Object* wrapDocument(WebCore::DOMDocument* doc);
  Runtime::Object* wrapElement(WebCore::DOMElement* element);
  ...
};

There is no corresponding .cpp implementation file.

More critically, the classes it tries to wrap do not exist in a usable form:

  • WebCore::DOMDocument — declared but not a real DOM tree
  • WebCore::DOMElement — no property storage, no live style resolution
  • No Node, Text, Comment, or DocumentFragment runtime types

Why This Breaks Modern Sites

Step in a React/Next.js page loadWhat should happenWhat actually happens
1. HTML loadsParser builds DOM treeMinimal DOM created (or none)
2. <script> runsZepraScript executes bundleExecutes in isolation — no DOM reference
3. React calls createRoot / renderBridge converts React elements → real DOM nodeswrapElement is a no-op or crashes
4. DOM mutations (appendChild, setAttribute, innerHTML)Layout + paint triggeredNothing to mutate — page stays empty
5. CSS cascade runsComputed styles appliedNo elements → no styles → nothing visible

The Missing Pieces (Implementation Debt)

  1. Concrete DOM classes — DOMElement, DOMText, DOMDocument with real childNodes, attributes, and ownerDocument.
  2. Bridge implementation — js_dom_bridge.cpp that actually allocates ZepraScript objects and installs getters/setters that forward to the C++ DOM.
  3. Global object wiring — window.document, document.createElement, document.body must exist before any user script runs.
  4. Mutation observation — layout and paint must be notified when JS mutates the tree (currently only the header promises this).

Impact on Beta Narrative

Public messaging must be honest:
“ZepraBrowser v0.1.0-beta renders static websites beautifully. JavaScript-heavy single-page applications (React, Next.js, Vue, Angular) currently render as empty shells because the DOM-to-JavaScript bridge is still being implemented. We are actively working on this for the next milestone.”

Recommended Path Forward

  1. Implement a minimal, read-only DOM stub (just enough for createElement, appendChild, textContent, setAttribute).
  2. Wire the bridge so that ZepraScript can obtain real object pointers instead of undefined.
  3. Add a simple test: load a tiny React “Hello World” and assert that the text node appears in the viewport.
  4. Only then re-enable full CSS/layout work — the CSS engine is already production-grade and should not be touched until the DOM layer exists.
